This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Lost Ark Astrogem cutting/fusion Monte Carlo simulator. Estimates success probabilities for achieving specific gem stat goals (e.g., min willpower 4 + min chaos 5 + boss_damage at level 5) across gem rarities, using the official in-game probability weights from Smilegate's published data.

## Running

Always activate the virtual environment before running Python:

```bash
source .venv/Scripts/activate
python -m arkgrid
```

No external dependencies for the simulator — stdlib only (`dataclasses`, `random`, `math`, `typing`). Vision features (`live`) require `opencv-python`, `numpy`, and `mss`. Automation (`auto`) additionally requires Windows (`ctypes` user32.dll). No build step, no linter configured.

## Architecture

The `arkgrid/` package is split into modules:

- **`constants.py`** — Gem effect definitions, coefficients, priorities, gem type mappings, fusion point-distribution constants (`FUSION_E_POINTS`), and `fusion_avg_coeff(gem_type, optimize, grade)` helper returning the expected side coefficient of a fusion-derived gem of the given grade
- **`models.py`** — `Option`, `LastTurnGoal`, `AstroGem`, `GemState`, `RunResult` (frozen/mutable dataclasses)
- **`pool.py`** — `OptionPool` — Builds the 27-entry weighted probability pool matching official rates; filters eligible options per turn based on current state and turn constraints (view and cost excluded on the last turn only — view CAN appear among the turn-1 picks, verified in-game; only the reroll *button* is locked on turn 1, and rerolls banked from a turn-1 view pick are usable from turn 2; cost eligibility tracks the live `cost_ratio` bounds); draws 4 unique offers via weighted sampling without replacement
- **`probability.py`** — `GoalProbabilityTable` — Precomputed DP table with four modes: standard (state = will, chaos, first, second, turns_left), reroll-aware (state adds reroll count), BIS-aware (state adds first_is_target/second_is_target booleans), and effect-aware (state adds first_idx/second_idx indices into `GEM_TYPES[gem_type]`). Transition probabilities use the single-draw approximation (option prob = weight/total eligible weight). The DP is stored in flat lists indexed arithmetically (`_s625`/`_idx`, layout `(state625, cs, ea, r, tl)` — same layout as the TS engine's `Float64Array`s); heavy effect-aware reroll builds still take seconds in CPython (no JIT), which is what `table_cache.py` amortizes. Every mode (both classes) additionally carries a 2-state **cost-saturation flag** `cs` in the DP key (right after the 4 stat levels): at `cost_ratio ±100` one weight-1.75 cost option leaves the pool (~1.8% probability shift onto everything else), and a cost pick toggles the flag (0→1 saturates, 1→0 returns to 0; the two saturated signs are symmetric, so one flag models cost exactly). Lookups read `state.cost_ratio`. Effect-aware + reroll-aware is the default and only DP mode used at runtime (self-disables only when no gem type is known). Per-offer posteriors go through `prob_after_option()` (`expected_prob_after_click` averages it over the hand); it is **destination-blind**: the change-effect card does not reveal its destination in-game, so change offers are valued as the average over the 2 non-equipped destinations and `Option.resolved_effect` is never consulted by decision metrics (the simulator still pre-resolves it, but only click *application* and offer logging use it). The reroll-aware mode (`max_rerolls > 0`) extends the DP with a per-option keep-vs-reroll decision at each state, using backward induction to find the optimal reroll timing. O(1) lookup. Supports side-node goals (min_first/min_second/min_side_coeff) via extended terminal success conditions. BIS-aware mode (`bis_only=True`) is a permanent no-op — effect-aware DP takes precedence and is always active when a gem type is known; `lookup_bis_averaged()` remains for compatibility but is not used on the hot path. Effect-aware mode (`effect_aware=True`, `gem_type`, `optimize`) tracks effect identity and models change_first/second_effect as probabilistic transitions to the 2 non-equipped pool members — correctly prices min_side_coeff goals when starting effects don't contribute to the target side. Early finish mode (`early_finish=True`) sets P=1.0 for any state where the goal is already satisfied (reflecting that the player can stop). `should_reroll_dp()` compares the expected value of keeping current offers against the value of rerolling (one fewer reroll, same state). `SideValueTable` is a parallel effect-aware DP whose value is the expected final gem value (`side_coeff + tier_bonus(total_points)`); it is consulted once the goal is met to decide finish-vs-continue. It is **reroll-aware** (`max_rerolls`, Phase A/B): the DP state carries the reroll count and the build prices a per-turn keep-vs-reroll choice via a variance-aware Gaussian `E[max(hand_EV, reroll_or_finish_value)]`, where `hand_EV ~ Normal(mu, (var/4)·fpc)` (the 4 offers are averaged; `fpc=(N-4)/(N-1)` is the finite-population correction over the `N` eligible pool members). This reroll-aware value drives **both** the finish-vs-continue decision gate and the displayed expected value, each threaded with the live reroll budget; `GemSimulator(reroll_aware_value=True)` is the default (`False` restores the legacy flat table). When `max_rerolls == 0` a reroll-aware build is byte-identical to a flat one. A second instance built with a trivial always-satisfied goal and no `min_side_coeff` floor — the **grade-value table** — is consulted on **dead-goal** turns (where the goal-conditioned table would zero every state). It is **always** built in `value_mode="grade_only"` (`tier_bonus` alone, side coefficients excluded), regardless of `--ignore-side-node-values`: a gem that missed its goal won't be equipped, so its side-node coefficients are worthless and only its fusion grade (relic/ancient) carries value. A dead goal therefore chases relic/ancient grade only and finishes the instant no higher grade (relic *or* ancient) is reachable — it never fishes for a side coefficient the dead gem can never cash in. `value_mode` has three values: `"side"` (default, `side_coeff + tier_bonus`), `"will_chaos"` (`will + chaos`, grade forced to 0), and `"grade_only"` (`tier_bonus` only). The web advisor's **displayed** eValue is decoupled from the decision table's `value_mode` and always reports an expected **side coefficient** (`engine/index.ts` `_displayValueTable`). Without `--ignore-side-node-values` it reads the reroll-aware `"side"` table (the value the decision already follows). Under `--ignore-side-node-values` the decision optimizes `will_chaos`, so a value-iteration `"side"` table would overstate the gem (it assumes the player chases the side node); instead the display is a **policy evaluation** of the will+chaos policy (`policy_value_mode="will_chaos"`, a coupled flat DP — `SideValueTable._build_policy_eval`/`_buildPolicyEval`), which follows the policy's finish-vs-continue choice and accumulates side value. So the shown eValue is the *realistic* expected coefficient — lower than the value-iteration figure, since the side node only rides along — and never the will+chaos sum (~4). Policy-eval is flat (no reroll dimension): the variance-aware reroll model exposes no discrete per-state argmax to follow, and under ignore rerolls aren't spent on the side node anyway. `SideValueTable` exposes public `relic_coeff` / `ancient_coeff` (the resolved tier-bonus values the gate reads); when no explicit value is passed it resolves the fusion-derived default (`constants.fusion_avg_coeff`), so grade is valued by default.
- **`table_cache.py`** — On-disk pickle cache for pre-built DP tables (stdlib only). `goal_table(...)` / `side_value_table(...)` are drop-in cached factories for the two table constructors; `simulator.py`, `automation.py`, and `cli.py` build all tables through them (imported as `cached_goal_table` / `cached_side_value_table`). Cuts CLI/auto startup from ~20s of builds to well under 1s once warm. Entries live under `%LOCALAPPDATA%/AstrogemCutter/dp-cache/<model-fingerprint>/` (Linux: `~/.cache/astrogem-cutter/...`), where the fingerprint hashes `probability.py`/`pool.py`/`constants.py`/`models.py` — any model change auto-invalidates, stale fingerprint dirs are pruned on first use, and the newest 128 pickles are kept (~2-9MB each; one `auto --all` sweep across the 6 gem types produces ~26, so a 32-file cap thrashed). Note the cache key includes `max_rerolls` — changing the reroll-budget sizing (rarity, ticket ownership, `--relic-reroll-threshold`) keys new entries, so the first run after such a change is a cold pass (~10-20s of builds per new gem type; `auto` prints per-table `[dp]` progress). Env knobs: `ASTROGEM_CACHE_DIR` (override root, used by tests), `ASTROGEM_NO_DISK_CACHE=1` (disable). Cache failures are never fatal (corrupt entries rebuild; failed writes skip caching).
- **`policy.py`** — `RerollPolicy` — **Logging-only utility.** Heuristic reroll policy retained for the `eff_threshold` log field in `simulate_one` and for `tests/test_policy.py` / `tests/test_scenarios.py`. The live decision path now goes through `arkgrid.decision`; do not add new decision logic here.
- **`decision.py`** — Shared per-turn decision logic for both the simulator and the automation loop. Pure functions, no state. Key types: `DecisionContext` (built once per run; bundles DP tables, goal, knobs, p_fresh, `confirm_active` / `confirm_min_coeff` / `side_value_table`, etc.), `TurnInput` (built per turn; state, offers, turn, turns_left, rerolls, reset_available, plus `ticket_lent` — whether `rerolls` includes the per-turn lent reroll ticket; free rerolls are spent first, so with `ticket_lent` and `rerolls == 1` a REROLL spends the gold-costing Charge, and reason wording says so instead of claiming a "free reroll"), `Decision` (action ∈ {PROCESS, REROLL, RESET, FINISH, FAIL} + branch tag + reason + metrics + `needs_confirmation` / `confirm_choices` fields). Six branch helpers (`_side_value_finish_decision`, `infeasibility_decision`, `prob_reset_decision`, `no_feasible_offer_decision`, `last_turn_reset_decision`, `dp_reroll_decision`) compose into `decide_post_roll(ctx, ti)`. **Free-reroll dominance**: every reset-capable branch (`_reset_or_chase_relic`, `prob_reset_decision`, `last_turn_reset_decision`) spends free rerolls that can still reach the goal (`p_reroll_goal > 0`) *before* the reset ticket — rerolling is free, never advances the turn, and unspent rerolls are lost on a reset anyway, so the reset fires only once rerolls are exhausted (or the goal is unreachable via any redraw). The pre-reset REROLL is never confirmation-gated (the gate guards the reset ticket; a free redraw spends nothing). Mirrored in the web engine (`decision.ts`), pinned by `tests/test_decision.py::TestRerollBeforeReset` and the `decisions.json` golden. The confirmation gate is wired in via `_side_value_finish_decision` (finish-vs-continue gate) and `_maybe_confirm` (stamps `needs_confirmation=True` and an action menu onto any eligible decision); `_side_coeff` and `_legal_actions` (which never offers REROLL on turn 1 — the game disallows it) are supporting helpers. `no_feasible_offer_decision` carries a **goal-met guard**: a gem that already satisfies the goal is a guaranteed success (early finish locks it in), so when every offer would break it irrecoverably (reachable only with an unknown gem type, where Branch 0 defers) the branch rerolls for a safe hand while a free reroll remains and finishes otherwise — it never hands a met goal to the reset/relic-chase tail. `feasible_count` (like all decision metrics) is **destination-blind** for change offers — see `prob_after_option` in probability.py. On dead-goal turns, `_reset_or_chase_relic` does **not** force-finish based on `--relic-reroll-threshold`: rerolls are free, so while a free reroll (or the current offers) can still reach relic+, the gem keeps chasing it — the threshold's only job is feeding the **per-turn extra-ticket gate** `ticket_enabled(ctx, state, turns_left, free_rerolls)` (see the *Extra ticket* concept below): the gold-costing ticket is re-evaluated every turn (never banked) and lent +1 reroll only when an enabler clears its bar. It delegates to `_grade_value_decision` (using `ctx.grade_value_table`, a **grade-only** table) to maximize expected gem **grade value** — pricing ancient (≥19) upside and point magnitude, which the binary `P(relic+ ≥ 16)` metric cannot. Side coefficients are excluded (a goal-missing gem won't be equipped), so it finishes early when neither a redraw nor the offers in hand can reach a higher grade (the one exception to "never finish with a free reroll" — there is provably no grade upside). When no grade table is present (gem type unknown), it falls back to the legacy binary relic+ chase. `_side_value_finish_decision`'s no-reroll branch implements the endgame-risk grade gate: `auto_gate = ctx.endgame_risk is None and not ctx.confirm_active` (`endgame_risk: Optional[float]`) — when auto-gating, a goal-met gem at relic/ancient grade whose side coefficient is below the grade benchmark (`ctx.side_value_table.relic_coeff` / `.ancient_coeff`) finishes to protect its grade; a float `endgame_risk` uses the player margin exactly — **also when the confirm gate is active** (the gate disables only the auto-gate, never an explicit margin). On goal-met turns at the will/chaos cap (`will==5 and chaos==5`) under `--ignore-side-node-values`, `_side_value_finish_decision` delegates to `_maxed_hold_decision`, which consults a side-mode "maxed oracle" (`ctx.maxed_value_table`) to chase `side_coeff + tier_bonus` while holding the cap firm — it never processes a hand containing a `will-`/`chaos-` offer (`_hand_is_wc_safe`), rerolls freely while upside remains, and finishes when the oracle sees no reachable upside. All relic+ overrides use offer-conditional posterior (`expected_prob_after_click`) — never the state prior. `compute_post_roll_metrics(ctx, ti)` runs the per-offer loop once so branch helpers don't repeat work. Adding a new branch: write `def my_branch_decision(ctx, ti, m) -> Optional[Decision]:` and insert it into `_POST_ROLL_BRANCHES` in priority order.
- **`simulator.py`** — `GemSimulator` — Turn-by-turn simulation engine. Manages rarity config (common=5 turns, rare=7, epic=9), reroll budgets (0/1/2 base + extra ticket), reset ticket (one full restart if goal becomes infeasible or probability drops below threshold), side-value finish (stop when goal is met per `SideValueTable` decision), and feasibility guards before each click. Per-turn decisions delegate to `decision.decide_post_roll`; the simulator handles offer generation, RNG state, and turn-log assembly around the call. Builds two DP tables: a reroll-aware table (for optimal reroll timing) and a standard table (for accurate reset decisions — the reroll-aware DP overestimates fresh-start probability due to the per-option max approximation). Always builds a relic+ DP table (`LastTurnGoal(min_total=16)`) when a gem type is known, for relic+ probability tracking and decision overrides (`--relic-reroll-threshold`). Also builds a `SideValueTable` for side-value finish decisions (lazily, cached per gem type), plus a goal-independent grade-value `SideValueTable` (`_get_grade_value_table`, cached per gem type) for dead-goal decisions. Both resolve `relic_coeff`/`ancient_coeff` from the fusion default when the flags are unset, and are built **reroll-aware** (`max_rerolls = self._dp_max_rerolls`) by default — the `reroll_aware_value=True` constructor flag; set it `False` to fall back to flat (reroll-unaware) value tables. `automation.py` builds the same value tables reroll-aware. Under `--ignore-side-node-values` it also builds a side-mode "maxed oracle" (`_get_maxed_value_table`, cached per gem type, `value_mode="side"`), used by the maxed-hold decision at the will/chaos cap. When the confirmation gate is active (`--confirm-min-coeff` set), on gated turns the simulator runs the recommended action but records a `confirm` marker in the turn log so gate behavior is auditable across `sim` batches. When `effect_aware=True`, builds per-gem-type effect-aware DP tables lazily (cached in `_ea_table_cache` / `_ea_reset_table_cache`) and swaps them in at `simulate_one` start — one table per gem type covers all effect configs, so `--all` and random-gem stats amortize the build across trials.
- **`automation.py`** — `run_auto()` — Full automation loop for the `auto` command: capture screen → detect state → decide (reroll/process/reset/finish) → click button → wait for animation → repeat. Windows-only (`ctypes` user32.dll for mouse clicks, focus check, Escape stop key). Per-turn decisions delegate to `decision.decide_post_roll` (same code path as simulator). Auto-detects gem type/effects from screen. Handles ticket confirmation dialogs and coordinate scaling for non-1080p monitors. When a decision has `needs_confirmation=True`, the loop prints a `[CONFIRM]` panel and blocks on a global F1-F4 hotkey menu (F1-F4 select from `confirm_choices`; Escape aborts the run); the user-selected action replaces the automated recommendation. `_build_prob_table()` maintains a module-level `_DP_CACHE` keyed on (goal, turns, gem_type, optimize, min_side_coeff, max_rerolls, early_finish) — in effect-aware mode a single cached table per gem type covers all effect configs, so `--all` mode rebuilds are eliminated for repeated gem types.
- **`analyzer.py`** — `GemAnalyzer` — Runs N trials (default 200k), aggregates success rate with Wilson confidence intervals, tracks relic+ (>=16 pts) and ancient (>=19 pts) thresholds, average side coefficient
- **`cli.py`** — CLI argument parsing and command handlers (`stats`, `sim`, `effects`, `live`, `auto`, `report`). Gem type auto-resolution from effect pairs in `_resolve_args()`.

Tests live in `tests/`, split by module (e.g. `test_pool.py`, `test_simulator.py`).

Developer scripts live in `tools/` (not part of the `arkgrid` package): `extract_templates.py` (crops template-candidate regions from screenshots into `tools/extracted/` for manual sorting, with a debug overlay per screenshot), plus `benchmark_reroll.py`, `calibration.py`, and `scenario.py` analysis scripts. Run them from the project root, e.g. `python tools/extract_templates.py`.

Requires `opencv-python`, `numpy`. Template-matching pipeline for recognizing the in-game astrogem cutting screen:

Two button-availability fields are read from **brightness** (not template matching, since the enabled/disabled glyphs are identical under a brightness-normalised matcher): `reset_enabled` (bright "Reset" text vs greyed) and `charge_enabled` (the yellow extra-reroll "Charge" button — bright fill ~0.08 of `ROI_CHARGE_BUTTON` vs ~0.00 when greyed; the button only shows "Charge" when free rerolls are exhausted, and the web ticket heuristic only consults `charge_enabled` there). ## Key Domain Concepts

- **Gem options**: willpower, chaos, first effect, second effect — each starts at 1, caps at 5
- **Side node goals**: `--min-first`/`--min-second` (level-based) and `--min-side-coeff` (coefficient-weighted total) add side node constraints to the success condition
- **Effect coefficients**: DPS: attack_power=400, additional_damage=700, boss_damage=1000. Support: ally_damage=600, brand_power=1050, ally_attack=1500
- **Cost ratio**: -100% to +100%, only changes via cost modifier possibilities
- **Rerolls** ("View Other Options"): re-draw all 4 offers; the base free rerolls (common 0 / rare 1 / epic 2) plus any earned from the pool, and the **reroll ticket** below
- **Reroll ticket** (once per *cutting process*): one extra "View Other Options" reroll. It is **not** a distinct "gold ticket" — it is just a reroll ticket that happens to be purchasable with in-game gold (you can buy more). CLI flag `--extra-ticket`, internal `use_extra_ticket`/`extra_ticket`.
- **Reset ticket** (once per *gem*): one full state reset to initial values, restarting the cutting process when the goal becomes impossible (or undesirable) mid-run. Because it starts a **fresh cutting process**, it re-enables that process's rerolls — both the base free rerolls **and** the per-process reroll ticket are granted again (e.g. epic = 2 free + 1 reroll ticket → reset → another 2 + 1).
- Each turn, one of 4 randomly presented possibilities is chosen uniformly (25% each)
- **Gem type auto-resolution**: `--first-effect` + `--second-effect` auto-resolves gem type. Same-type pairs (both DPS or both support) are always unambiguous. Three cross-type pairs need `--gem-type`: attack_power+ally_damage, additional_damage+brand_power, boss_damage+ally_attack
- **DP probability**: Analytical success probability from backward induction, shown alongside MC in `stats`. The DP has two layers: a reroll-aware table (state includes reroll count, used for optimal reroll timing via `should_reroll_dp()`) and a standard table (used for reset decisions and display). Both use the single-draw transition model (option prob = weight/total eligible weight). The reroll-aware DP uses a per-option keep-vs-reroll max in backward induction, which slightly overestimates reroll value, so the standard DP is used for reset decisions to avoid inflating the fresh-start probability. Effect-aware + reroll-aware is the default mode and correctly prices `--min-side-coeff` goals by tracking effect identity and modelling `change_effect` transitions; it self-disables only when no gem type is known. `--bis-only` is a permanent no-op — effect-aware DP takes precedence and is always active when a gem type is known.
- **`--trials 0`**: On `stats`, skips Monte Carlo and shows only the DP probability (instant).
- **Fusion-derived tier value**: `--relic-coeff` / `--ancient-coeff` default to the *fusion-derived average gem coefficient* for the run's gem type, not 0. `constants.fusion_avg_coeff(gem_type, optimize, grade) = pool_coeff_sum * E[points|grade] / 8` (E[points]: relic 16.25, ancient 19.05, from the official processed-fusion point distribution — value of fusing 3 same-type gems of that grade). Resolved inside `SideValueTable` when the flags are omitted; passing a flag (including `0`) overrides. This makes grade matter by default in both the side-value finish and the dead-goal decision.
- **Side-value finish** (`--endgame-risk F`): `_side_value_finish_decision` consults the (reroll-aware) `SideValueTable` DP at every goal-met turn to decide finish-vs-continue, threading the current reroll budget through the keep/process EVs so the value of an unspent reroll is priced in. `gem_value = side_coeff + tier_bonus(total_points)`, where `tier_bonus` adds `relic_coeff` for ≥16 points and `ancient_coeff` for ≥19 points (both resolved from fusion defaults or explicit flags). The decision never finishes while a free reroll remains (rerolls are free; gem supply is the bottleneck). Once rerolls are exhausted: **`--endgame-risk` passed** → finish when `finish_val >= process_ev + endgame_risk` (explicit margin); **`--endgame-risk` omitted (default)** → auto-gate by grade: a goal-met gem at relic/ancient grade whose side coefficient is below the fusion benchmark finishes to protect its grade; gems at/above the benchmark (or legendary grade) use margin 0 (EV-optimal). Gate-on (`--confirm-min-coeff N`) surfaces a side-value FINISH as an F1-F4 prompt for gems above the coefficient floor.
- **Relic+ tracking** (`--relic-coeff N`, `--relic-reroll-threshold F`): A separate DP table computes P(total_points >= 16) from the current state each turn; it always builds when a gem type is known. `--relic-coeff N` is the coefficient-equivalent worth of reaching relic+ grade (≥16 points), added to `gem_value` in the side-value DP; default is the fusion-derived average relic gem coefficient (`fusion_avg_coeff(gem_type, optimize, "relic")`); passing the flag overrides. `--relic-no-early-finish` is retired — its role is folded into `--relic-coeff`. `--relic-reroll-threshold F` is the **relic worthiness bar** for the reroll ticket and is one of the **per-turn reroll-ticket enablers** (`decision.ticket_enabled`): the ticket is lent on any turn where P(relic+) *with the ticket* (free rerolls + 1) ≥ F — goal-independent (grade is valuable regardless of the goal). It does **not** force-finish a dead gem — free rerolls always keep chasing a still-reachable relic+; the threshold limits only the reroll ticket's gold cost (the user's directive: "use free rerolls if it can reach relic+; the threshold is mainly there to limit the reroll-ticket usage"). Default 0.0 (disabled). P(relic+) is displayed in `sim` turn logs, `live` analysis, `auto` turn headers, and `stats` output whenever a gem type is known.
- **Ancient tracking** (`--ancient-coeff N`): `--ancient-coeff N` is the coefficient-equivalent worth of reaching ancient grade (≥19 points), added to `gem_value` in the side-value DP on top of any relic+ bonus; default is the fusion-derived average ancient gem coefficient (`fusion_avg_coeff(gem_type, optimize, "ancient")`); passing the flag overrides.
- **Dead-goal grade-value decision**: When the goal becomes unreachable, the per-turn choice (process / reroll / finish) maximizes expected gem **grade value** via a goal-independent grade-value `SideValueTable` built **grade-only** (`tier_bonus`, side coefficients excluded), instead of the binary `P(relic+ ≥ 16)`. This correctly prices ancient (≥19) overshoot and point magnitude (the binary metric scores totals 16/17/18/19 identically) — e.g. it locks in a relic+ gem on the last turn rather than gambling a forced click that could bust below 16, and it processes for a locked-in +3-to-ancient draw when ancient is richly valued. **It values grade only — never side coefficient — regardless of `--ignore-side-node-values`**: a gem that missed its goal won't be equipped, so its side nodes are worthless and only its fusion grade matters. So it **finishes the instant no grade upgrade (relic *or* ancient) is reachable** — redraw and offers both fail to beat stopping — even with rerolls left, instead of clicking on to pump a side coefficient the dead gem can never cash in. (While a grade upgrade *is* still reachable it keeps chasing it: a relic gem still continues for an ancient shot.) The grade-value table is built per known gem type in both `GemSimulator` and `auto` (coeffs resolve to the fusion default); the legacy binary relic+ chase runs only when the gem type is unknown. A reroll that can still reach the *goal* (Branch 3) takes priority over grade chasing. Separately, once will and chaos are both at the cap (5/5) the goal is locked, so the goal-met decision switches from the degenerate `will_chaos` value model to a side-mode oracle that chases side-node levels + grade as free upside, while holding will/chaos firm (only free rerolls and will/chaos-safe hands) and finishing when no further upside is reachable.
- **Confirmation gate** (`--confirm-min-coeff N`): Interactive pause for high-stakes decisions in `auto`. `--confirm-min-coeff N` is the sole confirm-gate knob — it activates the gate and prompts on every side-value FINISH for a gem whose side coefficient is above the floor. Two additional gates, also gated by `--confirm-min-coeff`: (1) **off-goal keep** — goal is structurally infeasible or no feasible offer exists, but the gem's side coefficient is valuable; (2) **reset** — auto would spend the reset ticket on a valuable gem. In `auto`, a gated turn prints a `[CONFIRM]` panel and blocks on a global F1-F4 hotkey menu (`confirm_choices` lists the selectable actions; Escape aborts the run); the chosen action replaces auto's recommendation. The simulator runs non-interactively — it executes the recommended action but records a `confirm` marker in the turn log on gated turns so behavior is auditable across `sim` batches. `--confirm-risk` is retired.
- **Forced reroll on no-progress turns** (`--force-reroll-no-progress N`): Coefficient-gated heuristic override to the DP reroll decision. When the gem's starting target-effect coefficient is ≥ N, force a reroll on any turn where no offer progresses the goal (no positive will/chaos delta, no positive side level when that side is needed, no side-coeff gain). Bypasses the DP's marginal keep-vs-reroll calculation, which would otherwise sometimes keep no-progress offers because of pool-dilution effects on future draws. MC shows +1-2.6pp on main-goal success at a cost of 1-5pp relic+ rate. 0 = disabled. Implemented in both `GemSimulator._has_progress_offer()` (wired into `roll_offers_with_rerolls`) and `automation._has_progress_offer()` (wired into the auto reroll block); per-run active flag is set based on the detected/configured gem's coefficient sum.
- **Automation** (`auto` command): Full automation loop — captures screen via `mss`, detects state via template matching, makes decisions using DP-optimal reroll timing + heuristic fallbacks, clicks buttons via `ctypes` user32.dll. Windows-only. Button positions at 1920x1080: Reset(962,255), Process(1068,765), Reroll(1254,595), Finish(831,764), Ticket confirm(906,666). Auto-detects gem type and effects from screen. `--dry-run` for testing without clicks. `--animation-delay` controls wait between actions (default 1.0s). Escape key stops automation. Pauses when Lost Ark loses focus. The post-action settle gate (`_still_waiting`) releases on a turn/reroll-counter change — or, after a Charge (ticket) reroll where neither changes, on a fully-detected new offer-card signature (`_offer_signature`), so the loop can't stall forever. End-of-gem JSONL success uses `_run_success` (goal + `bis_only` target effects + `min_side_coeff`, mirroring the simulator). DP table sizing and `ctx.base_rerolls` mirror the simulator exactly (rarity + owned ticket, +1 look-ahead headroom when the relic/goal ticket enablers are active). Tracks rerolls internally (OCR on new turns, decremented on rerolls) to handle view+N option picks.
- **Reroll ticket — per-turn gate** (`decision.ticket_enabled`): the **reroll ticket** (an extra reroll bought with in-game gold — not a separate "gold ticket"; CLI flag `--extra-ticket`, internal `use_extra_ticket`/`extra_ticket`) is **off by default** and **re-evaluated every turn (never banked)**. Each turn the ticket is *lent* +1 reroll to the decision budget only when the player owns it (tri-state `use_extra_ticket`/`extra_ticket`: `True`=force-on, `False`=hard-off, `None`=off-but-owned) **and** any enabler clears its bar — OR'd, each as a "with the ticket" look-ahead (`free_rerolls + 1`) except the reroll-independent coeff enabler:
  - `--extra-ticket` → always (force-on).
  - `--reroll-min-coeff N` → **expected** side-coefficient ≥ N, via a goal-conditioned expected-coeff table (relic/ancient coeff 0 ⇒ value = `E[side_coeff]`); it reads ~0 once the goal is unreachable, so a dead gem's worthless side coefficient turns the ticket off ("requires the goal" for free).
  - `--relic-reroll-threshold F` → P(relic+) with the ticket ≥ F (goal-independent).
  - `--reroll-goal N` + `--reroll-goal-threshold F` → P(will+chaos ≥ N) with the ticket ≥ F.
  Free rerolls are unaffected — always free to spend; the gate governs only the reroll ticket. The simulator/`run_auto` track it as a separate resource (the lend gate: `ticket_available` in the simulator, `extra_ticket_available` in `run_auto`), lend it per turn, and reconcile: it is **consumed** only when every free reroll *and* the reroll ticket were spent (free rerolls go first), otherwise returned. The reroll ticket is **once per cutting process** and **RENEWS on a reset** — the once-per-gem reset ticket starts a fresh cutting process, which re-enables both the free rerolls and the reroll ticket (e.g. 2 + 1 → reset → another 2 + 1), so the lend gate is re-armed at the new attempt; a NEW gem (`--all`) also re-arms it. The `ticket_consumed`/`extra_ticket_consumed` flag is the cumulative "was the reroll ticket ever spent this run" report (not the lend gate; never reset). `run_auto` reads the on-screen FREE reroll count (`extra_ticket=False`) and lends per frame. Using a reset or reroll ticket triggers an in-game confirmation dialog (automation clicks confirm at (906,666) after 0.5s; the reroll-ticket reroll fires only when free rerolls are exhausted — the "Charge" button).
- **Web extra ticket (stateless)**: the web advisor lends the ticket per *frame* (`computeAdvice` + `engine.ticketEnabled`), reading the on-screen free reroll count and adding +1 when enabled. It cannot track whether the ticket was already spent on an earlier turn (documented stateless limitation): `lib/app/ticket.ts` assumes the owned ticket is available while free rerolls remain, and once they hit 0 trusts the detected "Charge" button (`DetectionResult.chargeEnabled`, brightness-read like the reset button) when present. `--reroll-goal`/`--reroll-goal-threshold` are exposed in the web config (`rerollGoal`/`rerollGoalThreshold`, "Reroll goal" + threshold slider in the Advanced panel; the goal target blank = off, both knobs required). The action matrix shows **with vs without the extra reroll** side by side (`AdvisorOutput.ticket`) so the ticket's effect is visible. The detected-state panel shows the reroll budget split as `free(+1)/base` (e.g. `4(+1)/2`: detected free rerolls, `(+1)` while the ticket is lent this frame, `/base` the rarity's free-reroll grant), and the advisor panel warns when a recommended REROLL at 0 free rerolls spends the ticket (yellow Charge).
